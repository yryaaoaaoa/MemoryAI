# 高可用面试重点（故障防护 + 容灾 + 可观测性）

## 故障检测与主备切换
- 心跳检测：应用层（HTTP 探针）vs TCP 层，超时阈值与抖动容忍。
- 主备切换：主动-被动（冷备/热备）vs 主动-主动（双写+负载均衡）。
- 选主机制：Raft/Paxos 共识、ZooKeeper 临时节点、etcd Lease。
- 脑裂防护：Fencing（STONITH）/仲裁节点/ Lease 锁，避免双主写入。
- 切换窗口：RTO（恢复时间目标）与 RPO（恢复点目标）的工程实现。

## 限流、熔断与降级
- 限流算法：固定窗口（临界突刺）、滑动窗口、令牌桶（匀速放行）、漏桶（平滑输出）。
- 分布式限流：Redis Lua 原子计数 + 滑动窗口、Sentinel 集群限流模式。
- 熔断器模式：Closed→Open→Half-Open 状态机，失败率/慢调用率触发。
- 降级策略：返回缓存数据/默认值/简化结果、关闭非核心功能、手动开关。
- 级联故障预防：超时设置（connect/read）、重试次数与退避策略、舱壁隔离（线程池/信号量）。

## SLA/SLO 与可观测性
- SLA/SLO/SLI 关系：SLO 是目标（如 99.9% 可用）、SLI 是指标（延迟/错误率）、SLA 是合同。
- 三大支柱：Metrics（Prometheus + Grafana）、Logging（ELK/Loki）、Tracing（Jaeger/SkyWalking）。
- 告警设计：分级（P0-P3）、收敛（抑制/聚合/路由）、OnCall 轮值与升级。
- 错误预算：SLO 剩余预算决定发布节奏与变更风险容忍度。

## 容量规划与混沌工程
- 容量评估：QPS 峰值估算、带宽/存储/连接数上限、资源瓶颈定位（CPU/IO/网络）。
- 压测方法论：基准压测→阶梯加压→极限探测，关注长尾延迟（P99）而非均值。
- 自动弹性：HPA（Kubernetes）/ASG（云厂商），指标驱动（CPU/自定义指标）扩缩容。
- 混沌工程：Chaos Mesh/LitmusChaos，故障注入（网络延迟/节点宕机/依赖不可用）。

## 异地多活与容灾
- 同城双活：两个机房同时提供服务，数据库跨机房同步（延迟可控）。
- 异地多活：单元化架构（用户路由到固定单元）、数据同步（ Canal/Otter/Binlog）。
- 全球化部署：DNS 就近路由、CDN 静态资源加速、数据合规（GDPR 数据驻留）。
- 灾备演练：定期切换演练、数据一致性校验、回滚流程验证。

## 面试追问模板
- 流量突增 10 倍，你的系统哪里先扛不住？怎么应急？
- 熔断打开后，下游恢复了你怎么感知？自动恢复还是人工？
- 你的项目 SLA 是多少？怎么保证的？出过故障吗？
- 混沌工程你在团队推行过吗？遇到过什么阻力？
