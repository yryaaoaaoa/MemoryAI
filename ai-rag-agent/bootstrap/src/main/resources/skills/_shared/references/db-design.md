# 数据库设计面试重点（建模 + 索引 + 演进）

## 数据建模与范式
- 三范式核心：1NF（原子性）、2NF（消除部分依赖）、3NF（消除传递依赖）。
- 反范式设计：以冗余换查询性能，适用读多写少场景（如统计字段冗余）。
- ER 建模：实体→关系→属性，一对一/一对多/多对多的表结构映射。
- 宽表 vs 窄表：OLTP 倾向窄表（范式化）、OLAP 倾向宽表（星型/雪花模型）。

## 索引设计策略
- 索引选择性：高基数列优先，区分度 `COUNT(DISTINCT col)/COUNT(*)` 接近 1 最佳。
- 联合索引设计：等值条件列在前、范围条件列在后，遵循最左前缀。
- 覆盖索引设计：将查询所需的列全部纳入索引，消除回表。
- 避免过度索引：写入开销、存储成本、索引维护（DML 时需同步更新）。

## 冷热数据分层
- 冷热识别标准：访问时间衰减、业务状态流转（如订单已完成→归档）。
- 冷数据存储：归档表、历史库、对象存储（OSS/S3），压缩降低存储成本。
- 数据生命周期管理（DLM）：TTL 过期自动清理、定期迁移、分区裁剪。
- 分区表策略：RANGE（按时间）、LIST（按枚举）、HASH（按散列），分区裁剪加速查询。

## 扩容与迁移
- 垂直扩容（升级硬件）vs 水平扩容（分片/分库分表），各自的瓶颈与上限。
- 平滑扩容流程：新分片上线→双写→历史数据迁移→一致性校验→切读→下线旧分片。
- 数据迁移工具：Canal 监听 Binlog 增量同步、全量+增量切换方案。
- 回滚预案：切换失败时快速切回旧库，保留旧库只读窗口。

## 软删除与审计
- 软删除（`is_deleted` 标记）vs 硬删除：恢复能力 vs 存储膨胀与查询过滤。
- 审计字段：`created_at`/`updated_at`/`created_by`/`updated_by`，自动化填充（MyBatis 拦截器/JPA Auditing）。
- 数据版本控制：乐观锁版本号（`version`），防止并发覆盖。
- 敏感数据处理：加密存储、脱敏查询、合规删除（GDPR/个人信息保护法）。

## 面试追问模板
- 订单表数据量到亿级，你怎么设计索引和分片方案？
- 线上需要给大表加字段，怎么做不停机？
- 冷热数据迁移过程中，如何保证查询不受影响？
- 你的项目遇到过数据模型演进的问题吗？怎么处理的？
