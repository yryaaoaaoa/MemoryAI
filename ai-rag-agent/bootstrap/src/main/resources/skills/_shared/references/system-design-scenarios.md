# 系统设计与场景题（后端通用核心版）

## 1. 面试答题通用框架（先说这个）
1. 先澄清约束：QPS、峰值倍数、SLA、数据量、延迟目标、一致性级别、成本上限。
2. 再给核心链路：请求路径、读写路径、关键状态、失败路径。
3. 最后补工程细节：容量估算、监控告警、降级兜底、回滚与补偿。

## 2. 高频场景题的标准思路

### 秒杀系统
- 目标：抗瞬时洪峰、避免超卖、保证最终一致。
- 关键手段：前置限流 + 验证码/预约削峰 + 缓存库存 + 异步下单（MQ）。
- 一致性：预扣库存 + 订单超时回补；幂等键防重复下单。
- 风险点：热点 Key、缓存击穿、消息堆积、库存回补失败。

### 短链系统
- 目标：短码唯一、重定向高性能、可统计。
- 核心：短码生成（号段/哈希+冲突处理）+ 映射存储 + 缓存热点短链。
- 扩展：防刷、防爆破、过期回收、点击明细异步落库。

### Feed 流（推拉模型）
- 目标：低延迟、可扩展、支持热点用户。
- 模式：推模式（读快写重）/拉模式（写快读重）/推拉结合（明星与普通用户分治）。
- 关键：时间线有序、分页游标、防重复、防回刷穿透。

### 站内消息系统
- 消息类型拆分：系统通知、事件提醒、私信。
- 核心能力：未读计数、聚合展示、已读状态、离线补偿、幂等去重。
- 存储策略：会话维度和消息明细分离，冷热分层。

### 延时任务
- 可选方案：Redis 延迟队列 / MQ 延时消息 / 时间轮调度。
- 关键问题：任务可靠投递、重复执行防护、取消与重试语义。
- 工程点：失败重试上限、死信处理、任务可观测性（积压、延迟、失败率）。

### 登录风控（密码输错限制）
- 核心：用户维度失败计数 + TTL 锁定窗口。
- 要点：分布式一致计数、防账号枚举、风控阈值分级（验证码->临时锁定->人工校验）。
- 审计：保留关键操作日志与告警。

### 扫码登录 / 第三方登录
- 扫码登录：状态机（待扫码->已扫码待确认->确认登录/取消/过期）。
- OAuth2：授权码模式优先；短期 Access Token + 可撤销 Refresh Token。
- 安全点：`state` 防 CSRF、回调域名白名单、一次性 code、过期控制。

### 抢单 / 抢资源并发控制
- 核心目标：同一资源只被一个请求成功占有。
- 实现：数据库唯一约束 + 乐观锁版本号 或 Redis 原子指令/CAS。
- 必备：幂等键、超时释放、补偿回滚。

### UV/PV 统计
- UV 常见实现：HyperLogLog（省内存，近似去重）。
- PV：计数器即可；复杂场景按站点/页面/时间分桶。
- 大盘链路：写入削峰（MQ）+ 离线/准实时聚合。

### 去重推荐（大规模）
- 核心：布隆过滤器 + 分层存储（内存 + 磁盘）+ 批量异步刷盘。
- 重点：误判率控制、窗口淘汰策略、冷启动兼容。

### RPC 框架设计
- 关键组件：注册发现、序列化协议、网络传输、负载均衡、超时重试、熔断隔离。
- 高可用：请求幂等、降级策略、调用链追踪。

### Session 跨节点共享
- 推荐：集中式会话存储（Redis）。
- 不推荐：应用节点间 session 复制（扩展性差，维护成本高）。

## 3. 面试官常追问清单
1. 为什么这么分层？瓶颈先出在哪里？
2. 失败后怎么恢复？谁触发补偿？如何防重？
3. 峰值翻 3 倍时，先扩哪个组件？依据是什么？
4. 你怎么证明方案有效？给出核心指标与阈值。
5. 一致性和可用性冲突时，你的取舍原则是什么？
